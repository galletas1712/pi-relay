Design note

Steer promotion race: why the old error happened and what changed

The bug was not that steering was impossible. The bug was that the browser could show a stale queued follow-up row for a moment after the daemon had already materialized that row. Clicking steer during that window hit a backend error path meant for real invalid input. The fix has two layers: the backend now treats "no longer editable" as a normal no-op with a canonical queue snapshot, and the UI can replace queue state from events immediately so stale rows disappear faster.

The store treated "not currently editable" and "not found" as the same failure.

Queue row state machine

The important boundary is the transcript commit

Promotion is only a mutation while the row is in queued. New consumption keeps the row queued until the transcript transaction marks it consumed.

queued Editable. Can be promoted from follow-up to steer.
consuming Legacy claimed row from older daemons. No longer editable.
consumed Materialized into transcript and no longer in the visible queue.
cancelled Not currently exposed as a user action, but preserved as storage vocabulary.

What changed in code

File Responsibility Patch
agent-store/src/postgres/queue.rs Durable queue mutation Promotion now returns PromoteQueuedInputResult. If update matches no rows, it reads current row status before deciding whether this is a no-op or a real missing-id error.
agent-store/src/lib.rs Store-facing data model Added PromoteQueuedInputResult and simplified QueueMutationError to only mean not found.
agent-daemon/src/main.rs Websocket RPC input.promote_queued now returns input_id, priority, status, and promoted. It only publishes input.promoted when promotion actually happened.
agent-daemon/src/runtime.rs RPC error mapping The only queue mutation error now maps to input_not_found.
web/src/agentApi.ts Typed RPC client Typed the richer promotion response.
web/src/App.tsx User interaction Queue events patch the visible snapshot immediately: input.consumed removes rows and input.promoted marks rows as steer. After promote, the UI still refreshes session state; if promoted=false and the row is already past queued, it shows an info notice instead of an error.
rust/docs/websocket-rpc.md Contract documentation Documented the new no-op response for no-longer-editable rows.

The patch intentionally does not add retry loops or UI-side guessing. Postgres remains the source of truth: the client asks to promote a specific row, and the server replies with the row's durable outcome.
