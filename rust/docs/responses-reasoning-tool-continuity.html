Provider continuity surface

Reasoning and tool continuity across OpenAI and Anthropic

OpenAI Responses and Anthropic Messages can both be used in a stateless pi-relay architecture, but neither should be flattened to plain chat text. The common unit is a provider replay stream: visible messages, opaque reasoning/thinking continuity, tool calls, and matching tool outputs.

Short version

For pi-relay, "stateless provider" should mean: Postgres remains the canonical state, each provider call sends a reconstructed prompt/replay stream, and correctness never depends on provider-retained conversation state such as OpenAI previous_response_id.

That is valid, but only if we preserve the model output items needed to continue a tool loop. OpenAI calls these reasoning, message, and function_call items. Anthropic calls them content blocks such as thinking, redacted_thinking, tool_use, and tool_result.

This note is scoped to tools that participate in the model loop, including local/client tools and remote/provider tools such as web search. Cloud sandboxes, remote sessions, and managed cloud agents are intentionally out of scope.

Stateful continuation

Request 1 user input, tools
->
Response 1 OpenAI stores response state
->
Request 2 previous_response_id plus new input

This is simpler because the API can recover prior output items from the response ID. It is not the architecture we want as the source of truth.

Stateless continuation

Request 1 user input, tools, store:false
->
Persist output store typed output items ourselves
->
Request 2 replay previous output items plus tool output

This is the pi-relay shape: provider remains a pure adapter over our transcript store.

The common surface

The clean abstraction for pi-relay is not "Chat Completions messages" and not "Responses output items" directly. It is a provider-neutral transcript plus replay surface. The visible transcript can stay human-shaped, while the replay lane preserves provider-specific records byte-shape-stably enough to send them back.

Pi-relay concept OpenAI Responses Anthropic Messages Visible?
User input message with role:user role:user message content blocks Yes
Assistant text message with role:assistant, often with phase text content block in an assistant message Yes
Opaque reasoning continuity reasoning item with encrypted_content thinking block with signature, or redacted_thinking No, except optional debug metadata
Tool call function_call, custom/freeform call item, or provider tool call item tool_use block, or provider tool block such as server_tool_use Yes
Tool output function_call_output or provider tool result item tied by call ID tool_result user block, or provider result block such as web_search_tool_result Yes

Implemented provider-neutral shape

The core session FSM should not need to know whether replay data came from response.output[] or an Anthropic content block. It only needs semantic transcript items; provider-specific continuation payloads stay in an entry-aligned sidecar. 

struct ModelTranscriptEntry {
    item: TranscriptItem,
    provider_replay: Vec<ProviderReplayItem>,
}

struct ProviderReplayItem {
    provider: ProviderKind,
    raw_json: String,
}

The raw_json payload is what lets each provider serializer replay exact fields such as OpenAI encrypted_content, Anthropic signature, Anthropic redacted_thinking, remote tool fields, custom tool fields, and future provider-only metadata.

OpenAI continuity rule

The important rule from the reasoning guide is: when a reasoning model returns a tool call and we manually continue the conversation, preserve the relevant output items that led to that tool call and pass them back with the tool result.

For an agent, that means the replayed input after a tool completes should look like this:

message:user
The user's request, plus any earlier compacted context that is still model-visible.
reasoning
Opaque reasoning item returned by the model. In stateless mode, this needs encrypted content.
message:assistant
Visible assistant preamble or text, with its original phase preserved.
function_call
The exact tool call item returned by the model, including call_id, name, and arguments.
function_call_output
The result we produced by running the tool, tied to the same call_id.

The same idea applies to custom/freeform tools and remote/provider tools: preserve the returned call item shape and append or replay the matching output item.

What encrypted reasoning is

Reasoning tokens are not visible chain-of-thought. They are internal model computation. In ordinary API output, we do not get those tokens back. In stateless Responses mode, OpenAI exposes an opaque encrypted payload so the next request can continue the same reasoning/tool loop without revealing the reasoning text. 

{
  "model": "gpt-5.5",
  "store": false,
  "input": [...],
  "tools": [...],
  "include": ["reasoning.encrypted_content"]
}

The encrypted value is storage/replay material only. It should not be rendered in the UI, summarized, edited, or inspected.

Not thinking_redacted

This is not the old thinking_redacted row. That was a visible placeholder in our transcript. Encrypted reasoning is provider continuity state. It belongs in the model replay path, not in the user-facing transcript.

Why phase matters

GPT-5.5 and GPT-5.4 can produce assistant text before or between tool calls. OpenAI recommends preserving assistant message phase values in long-running or tool-heavy Responses flows. The two important values are:

Phase Meaning Why it matters
commentary Intermediate assistant update, such as "I'll inspect the logs first." Should not be treated as the final answer when replayed.
final_answer The completed user-facing answer. Marks that the model intended to finish the turn.

If we manually replay assistant history and drop phase, an intermediate preamble can look like a final answer to the model.

Anthropic continuity rule

Anthropic's Messages API has the same underlying shape, but the replay records live inside message content arrays instead of a top-level Responses output[]. Assistant messages can contain thinking, redacted_thinking, text, and tool_use blocks. Tool results are returned in the next user message as tool_result blocks.

With extended thinking and tool use, Anthropic says the full unmodified thinking block must be passed back during the active tool trajectory. The signature field is opaque verification material. A redacted_thinking block is also opaque and must be preserved if it is part of the active trajectory.

role:user
User asks for work.
thinking
Assistant thinking block, including exact signature. Replay-only.
text
Optional assistant preamble or visible progress text.
tool_use
Tool request with id, name, and JSON input.
tool_result
Next user message content block with the same tool_use_id.

Different placement, same invariant

OpenAI returns replay records as response output items. Anthropic returns replay records as message content blocks. The invariant is the same: do not modify, reorder, orphan, or partially drop the active reasoning/thinking -> tool call -> tool result chain.

Anthropic request shape

The harness in ~/Downloads/src builds Anthropic requests around anthropic.beta.messages.create. It sends normalized messages, system blocks, tools, tool_choice, optional thinking, beta headers, prompt-cache breakpoints, and optional API context management. 

{
  "model": "claude-...",
  "system": [...],
  "messages": [
    {
      "role": "assistant",
      "content": [
        { "type": "thinking", "thinking": "...", "signature": "..." },
        { "type": "tool_use", "id": "toolu_1", "name": "read", "input": {...} }
      ]
    },
    {
      "role": "user",
      "content": [
        { "type": "tool_result", "tool_use_id": "toolu_1", "content": "..." }
      ]
    }
  ],
  "tools": [...],
  "tool_choice": { "type": "auto" },
  "thinking": { "type": "enabled", "budget_tokens": 12000 }
}

OpenAI request shape

The equivalent stateless OpenAI request sends the complete reconstructed input and asks for encrypted reasoning content when reasoning continuity matters. 

{
  "model": "gpt-5.5",
  "instructions": "...",
  "store": false,
  "include": ["reasoning.encrypted_content"],
  "input": [
    { "type": "message", "role": "user", "content": [...] },
    { "type": "reasoning", "id": "rs_...", "encrypted_content": "..." },
    { "type": "function_call", "call_id": "call_1", "name": "read", "arguments": "{}" },
    { "type": "function_call_output", "call_id": "call_1", "output": "..." }
  ],
  "tools": [...]
}

What the Anthropic harness teaches us

~/Downloads/src is complex, but the useful lessons are concrete:

  • Tool result pairing is structural. The harness normalizes and validates tool_use / tool_result pairing before calling the API because orphaned results or duplicate IDs cause hard provider errors.
  • Thinking blocks are protected replay records. The harness comments state that thinking/redacted-thinking blocks must survive the active assistant trajectory. It also tombstones partial streaming messages because invalid thinking signatures can poison future requests.
  • Streaming fallback is dangerous around tools. If a partial stream already emitted tool calls, a fallback retry can produce different tool IDs. The harness discards pending results from failed attempts to avoid orphaned tool outputs.
  • Prompt caching is prompt-shape-sensitive. The harness caches tool schemas, applies cache_control at selected system, tool, and message boundaries, and avoids mutating assistant messages that will be replayed.
  • Provider-specific cleanup should live at the boundary. Claude Code has many normalizers for Anthropic-only fields. Pi-relay should keep that kind of provider translation inside agent-provider, not in the core FSM.

Lossy OpenAI replay

This is close to our current Chat-Completions-shaped transcript. It keeps only visible text, the tool call, and the tool result. 

[
  { "role": "user", "content": "Fix the bug" },
  { "role": "assistant", "content": "I'll inspect files." },
  {
    "type": "function_call",
    "call_id": "call_1",
    "name": "read",
    "arguments": "{\"path\":\"src/lib.rs\"}"
  },
  {
    "type": "function_call_output",
    "call_id": "call_1",
    "output": "file contents..."
  }
]

Missing: encrypted reasoning and assistant phase. It may work sometimes, but it is lossy. The Anthropic analog is dropping thinking.signature or redacted_thinking.

Faithful OpenAI replay

This keeps the response output items that the reasoning model needs to continue the tool loop. 

[
  { "role": "user", "content": "Fix the bug" },
  {
    "type": "reasoning",
    "id": "rs_...",
    "encrypted_content": "..."
  },
  {
    "type": "message",
    "role": "assistant",
    "phase": "commentary",
    "content": [{ "type": "output_text", "text": "I'll inspect files." }]
  },
  {
    "type": "function_call",
    "call_id": "call_1",
    "name": "read",
    "arguments": "{\"path\":\"src/lib.rs\"}"
  },
  {
    "type": "function_call_output",
    "call_id": "call_1",
    "output": "file contents..."
  }
]

This is still stateless: everything needed for the next request is stored by pi-relay.

Faithful Anthropic replay

The Anthropic equivalent is not a top-level item list. It is an assistant message whose content blocks preserve thinking and tool use, followed by a user message whose content starts with the matching tool result. 

[
  {
    "role": "assistant",
    "content": [
      {
        "type": "thinking",
        "thinking": "...",
        "signature": "opaque verification string"
      },
      {
        "type": "text",
        "text": "I'll inspect files."
      },
      {
        "type": "tool_use",
        "id": "toolu_1",
        "name": "read",
        "input": { "path": "src/lib.rs" }
      }
    ]
  },
  {
    "role": "user",
    "content": [
      {
        "type": "tool_result",
        "tool_use_id": "toolu_1",
        "content": "file contents..."
      }
    ]
  }
]

Anthropic's docs also allow redacted_thinking. Treat it like encrypted reasoning: store it, replay it, do not render it as useful content.

Implications for pi-relay

  • Provider stays stateless. Each provider can build a complete request from our transcript and replay lane. OpenAI does not need previous_response_id; Anthropic does not need hidden session state in the Messages API.
  • Transcript needs a provider replay lane. We need somewhere to persist replay-only OpenAI output items and Anthropic thinking blocks without rendering them as chat messages.
  • Active tool loops should not be compacted away. Compaction can summarize completed prior history, but it should not rewrite or drop the open chain of reasoning/thinking -> tool call -> tool output.
  • Fork and switch should branch replay state too. If a user branches from a point inside a provider tool loop, the branch must include the replay records that are ancestors of that point.
  • UI should render only human-meaningful items. Text, tool calls, tool results, crashes, compactions, and branches are visible. Encrypted reasoning, Anthropic thinking signatures, and redacted thinking are not.

What can go wrong if we drop items

The model may treat an intermediate preamble as a completed answer, fail to connect a tool result to the reasoning that requested it, repeat a tool call, lose track of whether it was in a planning/commentary phase, reject a request because a thinking signature changed, or behave differently after a fork/compact even though the visible transcript looks equivalent.

Useful design rule

Store visible transcript items for humans, and store provider replay items for the model. Those two lanes can reference the same logical turn, but they should not be forced to have the same shape.
