pi-relay design explainer

Workflow orchestration without the orchestration tax

The new plan replaces ad hoc Python REPL subagent orchestration with discoverable workflow plugins, scoped task agents, durable artifacts, and a simple run board UX.

4 primitives workflow plugins regular tool calls scoped task agents artifacts as handoffs

Quick map

01 · What changed

From subagent scripts to workflow plugins

The old mental model asked a parent agent to write Python control flow against subagents. The new model gives top-level agents normal tools to discover and invoke workflow plugins.

Old path: parent-agent orchestration

  • Parent model writes Python REPL orchestration.
  • Subagents are the primary abstraction.
  • Control flow lives in ephemeral Python state.
  • Handoffs are transcript snippets or manual summaries.
  • Human interruption makes parent control flow fragile.

New path: workflow plugins

  • Top-level model calls workflow.list and workflow.start.
  • Run/Task/Artifact are the durable abstractions.
  • Control flow lives in plugin code and persisted state.
  • Artifacts are the default handoff mechanism.
  • Task agents are scoped, observable, and interruptible.
02 · The model

Four primitives carry the whole system

The plan intentionally avoids extra tables for proposals, budgets, variables, and leases. If it can be a task result or an artifact, it is not a new abstraction.

1

Run

Overall goal, plugin kind, status, params, and small plugin-owned state.

2

Task

One unit of work, usually a scoped child agent session with a role and prompt.

3

Artifact

Handoffs, evidence, diffs (incl. code changes), logs, snapshots, decisions, notes.

4

Workspace source

How a task sees files: no workspace, read parent, fork parent, or from one prior task. Single ancestor only.

03 · Plugin registry

Workflows are discoverable plugins

"Plugin" means the top-level agent can discover the workflow, read its input schema, and start it through regular tool calls. It does not mean we need a generic graph DSL in v1.

Manifest

{
  "id": "kubernetes_e2e",
  "title": "Kubernetes e2e test",
  "description": "Run adaptive tests against a cluster.",
  "input_schema": {
    "required": ["context", "namespace", "test_goal"]
  },
  "default_roles": ["kubernetes-tester"]
}

Runtime contract

trait WorkflowPlugin {
    fn manifest(&self) -> Manifest;
    fn start(&self, params: Json) -> Advance;
    fn advance(
        &self,
        run: Run,
        tasks: Vec<Task>,
        artifacts: Vec<Artifact>,
    ) -> Advance;
}
04 · Human UX

The human sees a run board

The UI should be about what is happening and what needs attention, not about graph topology. Most actions are verbs on task cards and human requests.

Run 1

Kubernetes e2e Goal: test the operator on dynamo-nscale-dev. running
Current phase Diagnosing rollout and image version.

Task cards 3

kubernetes-tester Collecting pod events and rollout status. running
evidence artifact pod-describe, events, image digest. written
human request Teleport auth renewal needed. blocked

Controls verbs

Available actions Pause · cancel · retry task · steer child · answer human request
Latest decision Suggested next: rebuild image, then rerun rollout.
05 · Agent UX

The agent receives a scoped task contract

Workflow task agents are not little orchestrators. They receive a role, selected context, workspace instructions, allowed outputs, and a stop condition.

Example task prompt

You are task task_k8s_1 in run run_42.

Role:
  kubernetes-tester

Run goal:
  Run e2e tests for the operator.

Task goal:
  Deploy the test manifest, observe rollout,
  collect evidence, classify the outcome.

Context:
  artifact_build_image_digest
  artifact_test_manifest

Workspace:
  read_parent

Allowed outputs:
  complete with one of:
    pass | product_failure |
    environment_retry | human_needed

Stopping condition:
  Stop when tests pass, a code issue is found,
  or human action is required.

Example task result

{
  "status": "done",
  "summary": "Rollout failed because the deployed image digest is stale.",
  "artifact_id": "artifact_k8s_rollout_evidence",
  "suggested_next": "rebuild_image"
}

The workflow plugin reads suggested_next and decides whether to create another Kubernetes tester task, ask for a rebuild, or mark the run failed.

06 · Examples

What actual exchanges look like

These examples show the system from the perspective of a user, the top-level agent, workflow tools, and task agents.

Kubernetes e2e: user asks for adaptive cluster testing
User Run the operator e2e tests in dynamo-nscale-dev, namespace schwinns. If it is an infra issue, adapt and retry. If it is a code issue, stop with evidence.
Top-level agent I’ll look for an appropriate workflow plugin and start it with the cluster details.
Tool callworkflow.list()
Tool result Available: explore, implement_review_test, kubernetes_e2e, parallel_race.
Tool callworkflow.describe({ "id": "kubernetes_e2e" })
Tool callworkflow.start({ "kind": "kubernetes_e2e", "goal": "Run operator e2e tests", "params": { "context": "dynamo-nscale-dev", "namespace": "schwinns", "test_goal": "operator e2e", "stop_when": "pass, code issue found, or human needed" } })
Workflow Created run run_42. Started task task_k8s_1 with role kubernetes-tester.
Task result task_k8s_1 wrote evidence and completed: suggested_next = "rebuild_image". Summary: deployed digest does not match current code.
Top-level agent The run found a code/deployment issue, not a cluster scheduling issue. The evidence artifact includes rollout status, pod events, and image digest.
Implement/review/test: reviewer loop before testing
User Implement this change, get a reviewer happy, then run tests. If tests fail, go back through review before testing again.
Tool callworkflow.start({ "kind": "implement_review_test", "goal": "Implement durable run/task/artifact tables", "params": { "acceptance": "tables, repo methods, basic tests", "test": "cargo test -p agent-store", "stop_when": "reviewer approves and tests pass" } })
Workflow Round 1: started implementer. Its handoff artifact becomes the input to reviewer.
Reviewer task result Changes requested: migration is missing an index. Suggested next: implement_changes.
Workflow Round 2: starts implementer from previous implementation artifact and reviewer notes. No testing yet.
Reviewer task result Approved. Suggested next: test.
Tester task result Tests failed because repo method returns stale status. Suggested next: code_issue.
Top-level agent The workflow is returning to the implement/review loop with the test evidence attached.
Explore: parallel RLM-style exploration
User Explore options for making workflow plugins load from workspaces. Give me tradeoffs.
Tool callworkflow.start({ "kind": "explore", "goal": "Explore workspace-loaded workflow plugins", "params": { "question": "What plugin formats and trust model should we use?", "num_explorers": 4 } })
Workflow Spawned 4 explorer tasks. Each receives a different angle: Rust ABI, WASM, manifest-only plugins, and security/trust.
Reducer Synthesized explorer artifacts into one decision artifact with recommended v1/v2 split.
Hill-climb: propose/evaluate candidates
User Find the simplest API shape for workflow plugin manifests. Iterate until the evaluator thinks it is simple enough.
Workflow Iteration 1 proposer writes candidate manifest. Evaluator scores clarity 6/10 and asks to remove lifecycle hooks.
Workflow Iteration 2 proposer simplifies to id/title/description/input_schema/default_roles. Evaluator scores 9/10. Best artifact updated.
07 · Subagents

Subagents become execution modes, not the orchestrator

Workflows reduce the need for recursive subagent orchestration, but we should not remove async/observable helper agents entirely.

Mode Used for Properties Tools
Workflow task agent Tasks created by workflow plugins. Scoped, non-recursive, async, observable, interruptible, artifact-producing. task.write_artifact, task.complete, task.fail
Inline call Small read-only summarization/review/classification tasks. One-shot optimization; no workspace write; no separate long-lived control flow. Task result only, optionally artifact write.
Direct delegation agent Ad hoc top-level splitting outside a formal workflow. Async, inspectable, cancellable/steerable, optional isolated workspace. agent.spawn, agent.status, agent.read, agent.cancel
Design rules

How to keep it small

Prefer artifacts over abstractions

If a concept can be a handoff, evidence file, diff, snapshot, or decision artifact, do not add a table.

Prefer plugin code over graph DSL

If control flow can live in a workflow plugin's advance function, do not create a generic graph edge.

Prefer scoped agents over recursive agents

A workflow task agent should complete its task, write artifacts, and stop. The plugin creates more tasks if needed.

Prefer normal tool calls

Top-level agents discover and invoke workflows with workflow.*. Python REPL remains an escape hatch.
