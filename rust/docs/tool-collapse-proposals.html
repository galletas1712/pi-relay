Consecutive Tool Collapse Proposals

The transcript currently renders each tool call as an equal-weight card. That is useful while debugging, but long coding turns become visually noisy. The cleanest model is to derive collapsible tool groups in the web UI from existing transcript entries, without changing Postgres or the core session model.

Recommended Direction

Group consecutive tool activity into a derived ToolRunGroup in the frontend. Completed successful read/search/bash runs should collapse into a compact summary. Active, failed, interrupted, and edit/diff work should stay visible enough to explain what is happening.

This keeps the persisted transcript append-only and literal, while letting the UI present the transcript at a humane density. Collapsing is only a default view. There must always be a path to expand back to the exact underlying transcript entries.

Design A: Completed Group Summary

After a turn finishes, consecutive successful tool calls collapse into one row with counts and the last meaningful labels. Errors and edits can either be pulled out or kept expanded inside the group. The group can still be expanded to reveal every individual call.

Local Disclosure

The primary interaction should be local disclosure: expand one group, inspect it, then collapse it again. There is no separate raw transcript mode in this proposal. Full detail is available by expanding the relevant group in place.

State What It Shows Why It Exists
Collapsed group Grouped summary, status, counts, and high-signal labels. Everyday reading after a long turn.
Expanded group Every tool call/result inside that group, including repeated searches, intermediate successes, and any future operational entries that belong to the group. The inspection path. Users usually care about one burst of work at a time.

Recommended default: only one tool group is expanded at a time inside the current turn. Opening a second group closes the previous expanded group, unless the user pins a group open later.

How It Should Behave Mid-Turn

The active turn should not hide too aggressively. The user needs to know whether the agent is stuck, exploring, editing, or validating. A good live rule is:

Tool State Live Rendering After Turn Completes
Currently running Always visible as the last row in the active group. Fold into the group if successful; remain visible if failed/interrupted.
Recent successes Show the latest 2-3 rows so the user can see motion. Collapse into a count summary.
Older successes Collapse immediately into “N earlier tools”. Stay collapsed.
Errors Always visible, expanded enough to show the error line. Stay visible unless manually collapsed.
Edits Show a compact “Edited file +N -N” row, expandable to diff. Keep as a visible edit summary; collapse raw metadata.

Edit And Diff Treatment

Edits should not rely on raw tool JSON. Both OpenAI apply_patch and Claude text editor calls can render as the same pretty name, Edit, then show an edit summary and optional diff.

Edited transcript.tsx +3 -2
-42prettyName: "str_replace_based_edit_tool"
+42prettyName: "Edit"
-91renderRawToolMetadata(argsJson)
+91renderDiffSummary(display.diff)

Implementation Shape

This can be a pure presentation layer at first. Keep TranscriptEntry unchanged. Derive grouped display nodes after branchEntriesFor(...) and before rendering.

Layer Responsibility
Rust session/store No change. Continue storing literal assistant messages, tool calls, results, and provider replay.
Rust provider/tools Keep emitting stable pretty names and input summaries. Later, optionally attach structured diff metadata for edit tools.
Web turn view Build derived ToolRunGroup nodes from consecutive tool render parts.
Web renderer Render active groups expanded with recent rows; render completed groups collapsed by default, preserving errors and edit summaries.

Recommended Rules

  1. Only group consecutive tool calls that have no assistant text between them.
  2. Start a new group across user messages, turn boundaries, compaction summaries, or assistant text.
  3. During a running turn, keep the active group expanded and show the running tool plus the latest successful tools.
  4. After a graceful turn, collapse successful read/search/bash groups by default.
  5. Never auto-hide failed, interrupted, or crashed tool calls.
  6. Render edits as Edited file +N -N with an expandable diff, not raw JSON.
  7. Remember manual expand/collapse per group id while the session is open.
  8. Prefer one expanded group at a time; optionally allow pinning multiple groups open later.
