Assistant Boundaries, Turns, and Branching

Plan for removing pair-shaped assumptions from fork, switch, export, and rendering. The core idea is simple: a user message is not paired with exactly one assistant message. A turn is a sequence of user inputs, model steps, tool work, and a terminal boundary.

Current risk pair-shaped UI

Some UI/export paths treat each assistant message as the answer to the latest user message. That breaks down when providers emit text before tool calls, then later emit a final answer.

Durable invariant append-only tree

The transcript forest is already the right foundation. Branching should target entries and stable turn boundaries, not inferred user/assistant pairs.

Design move turn groups

Build a derived turn view over the transcript path. Each assistant message becomes a model step whose phase is inferred from neighboring transcript entries.

What the Providers Actually Return

Both OpenAI Responses and Anthropic Messages can produce assistant text before tool calls. That text is not necessarily the final answer. The provider response is better understood as one model step: "the model advanced until it either ended, requested work, paused, or hit a limit."

ProviderUser input shapeAssistant output shapeProvider signal
OpenAI Responses Input is an item list. A user message is a message item with role: "user" and typed content parts. Output is an item list. A single response can contain assistant message items, function_call items, reasoning items, and built-in tool items. There is no separate "intermediate" or "final summary" item type. If the output includes client tool calls, the runtime continues through tools. If it has no client tool calls, core finishes the pi-relay turn.
Anthropic Messages Input is a stateless message list. User messages contain text/image blocks and, after client tools, tool_result blocks. Assistant messages contain content blocks such as text, thinking, redacted_thinking, and tool_use. stop_reason is explicit: tool_use, end_turn, max_tokens, pause_turn, etc. Pi-relay can use this while parsing, but it does not need to persist it as transcript truth.

OpenAI's Responses API is item-oriented, so it already matches an append-only transcript better than Chat Completions did. The important caveat is that an assistant text item is not self-labeling as "intermediate" or "final"; pi-relay should derive that from transcript structure after the provider result has been applied.

Anthropic folds tool results into the user/assistant message structure instead of using a separate tool role. That is a provider serialization detail, not a reason for pi-relay's core transcript to pretend tool results came from the human user.

Vocabulary

TermMeaningWhy it matters
Turn A contiguous span from TurnStarted to TurnFinished, or from a compaction root to the next boundary. The natural unit for switch, retry, continue, and most export choices.
User input Initial user message, follow-up, or steer inserted into a turn. There may be several user inputs inside a single turn.
Model step One provider call result, stored today as AssistantMessage. A turn may have many model steps: tool request text, tool calls, final answer.
Model content Text, tool calls, images later, and provider-hidden continuity blocks carried by a model step. This keeps "assistant message" from implying "final answer".
Model step phase A derived UI/export label such as tool request, final answer, running, aborted, or unknown. This should be computed from transcript neighbors, not persisted on AssistantMessage.
Provider stop signal Provider-local metadata such as Anthropic stop_reason or OpenAI response incomplete status. Useful while parsing current output, but not the durable boundary model.
Stable boundary Empty root, TurnFinished, or CompactionSummary. Switch/rewind should only activate stable boundaries.

Middle-Ground Abstraction

The clean middle ground is not to make pi-relay OpenAI-native or Anthropic-native, and not to add a new durable "continuation" enum. It should keep the transcript entries we already understand and derive richer UI/export views from the transcript path.

LayerOwnsDoes not own
Provider adapter Serialize pi-relay context into OpenAI items or Anthropic messages; parse provider output into model content, tool calls, usage, replay blocks, and provider-local stop/error metadata. Fork/switch rules, UI grouping, or transcript tree mutation policy.
Core FSM Append user inputs, model steps, tool results, turn boundaries, crash/interruption outcomes, and compaction summaries. Provider-specific item names like function_call_output or Anthropic block ordering rules.
Session/transcript store Append-only tree/forest, active leaf, provider replay sidecars, and stable source mutations. Final-answer inference or visual grouping.
Web UI Derived TurnView grouping, picker labels, export shape, rendering weight, and command UX. Provider wire schemas or durable provider continuity contracts.
The abstraction should be rich enough to represent many user inputs and many model steps per turn, but small enough that old sessions still load with no data migration. The durable source of truth remains the append-only transcript.

Current Assumptions to Remove

AreaObserved assumptionReplacement
Export buildExportBlocks tracks the most recent user entry and assigns it to every later assistant block. Export turns or model steps with their containing turn context. Include all user inputs since the previous boundary when exporting a final answer.
Switch picker Turn boundary rows preview only the nearest previous assistant message. Preview the terminal model step for the turn, but label it as a turn boundary. Show tool-request/progress model steps only as context rows.
Fork picker Every assistant entry is offered as an "assistant message" target without saying whether it requested tools or ended the turn. Label assistant targets as model steps: "after tool-request step", "after final answer", or "after model step".
Rendering All assistant text is rendered with the same visual weight. Render tool-request/progress text as context and final-answer text as the answer. Tool cards stay attached to the model step that requested them.
Core naming AssistantMessage sounds like a conversational answer. Treat it internally as a model response step. Rename later only if it pays for itself.

Plan

Phase 1: Keep the durable transcript model simple

  1. Do not add a durable ModelContinuation enum to AssistantMessage.
  2. Keep the current transcript vocabulary: UserMessage, AssistantMessage, ToolCallStarted, ToolResult, TurnFinished, and CompactionSummary.
  3. Let core continue using the existing rule: assistant tool calls request tools; assistant messages without tool calls finish the turn gracefully.
  4. Provider-local stop details can be used to produce better errors or retries, but they should not become the durable app boundary model until there is a real recovery feature that needs them.
  5. Keep thinking/reasoning data in provider replay, not visible assistant text.

Phase 2: Enforce Rust-owned structural invariants

  1. Rust should reject source mutations that target invalid switch boundaries. Switch remains root, TurnFinished, or compaction root/summary only.
  2. Rust should reject fork targets that are not root, TurnFinished, or CompactionSummary.
  3. Rust should preserve atomic active-leaf movement when appending transcript rows, switching, forking, compacting, or consuming queued input.
  4. Rust should continue ensuring that model tool calls become ToolCallStarted rows before tool results are appended.
  5. Rust recovery should close open non-live tails as Crashed, while explicit user stop closes as Interrupted.
  6. Rust should keep provider replay sidecar-only; replay records must not appear as visible transcript content.

Phase 3: Build a derived turn-view module

  1. Create a pure helper in the web UI that converts branch entries into TurnView[]. Do not mirror this helper in Rust unless a backend feature needs the derived presentation shape.
  2. Each TurnView contains user inputs, model steps, tool results, terminal outcome, and stable boundary id.
  3. Each ModelStepView derives its phase from transcript structure: tool request, final answer, running, aborted, or unknown.
  4. This helper must tolerate malformed or open turns by marking them incomplete rather than inventing a pair.

Phase 4: Make fork/switch/export consume turn views

  1. Switch remains stable-boundary only: empty root, compaction root, or TurnFinished. Editing a user message still maps to the previous stable boundary plus restored composer text.
  2. Fork can still target any entry, but the picker should describe the target accurately: before user input, after model step, after tool result, or after turn boundary.
  3. Export defaults to final answers or whole turns. Advanced export can include tool-request/progress model steps and tool traces.

Phase 5: Add regression fixtures

  1. Fixture: assistant text plus tool call, tool result, final assistant text.
  2. Fixture: two steers in one turn, producing multiple user messages and multiple assistant steps.
  3. Fixture: fork from a tool-request/progress assistant model step closes the open turn as interrupted in the fork only.
  4. Fixture: export final answer includes the relevant turn context without duplicating or dropping user inputs.
  5. Fixture: switch never activates a non-boundary model step.

Data Shape Sketch

// Durable model stays unchanged.
struct AssistantMessage {
  items: Vec<AssistantItem>,
}

// Derived by UI/export helpers from a branch path.
enum ModelStepPhase {
  ToolRequest,
  FinalAnswer,
  Running,
  Aborted,
  Unknown,
}

type TurnView = {
  turnId: number | null;
  startEntryId: string | null;
  boundaryEntryId: string | null;
  outcome: "Graceful" | "Interrupted" | "Crashed" | null;
  userInputs: TranscriptEntry[];
  modelSteps: Array<{
    entry: TranscriptEntry;
    phase: ModelStepPhase;
    toolCalls: ToolCall[];
    text: string;
  }>;
  toolResults: TranscriptEntry[];
};

This requires no transcript migration. Existing sessions load exactly the same; the helper infers phase from tool calls, tool-start/result entries, open session activity, and nearby TurnFinished boundaries.

Why This Does Not Blow Up Complexity

  • No durable field is added to assistant transcript items.
  • The transcript store remains append-only and provider-agnostic.
  • The provider adapters keep their exact replay sidecars for OpenAI/Anthropic continuity, but the rest of the system reads normalized transcript items.
  • The richer concepts, such as "final answer for this turn", are derived by a pure helper over the active branch path.
  • Old sessions degrade gracefully: assistant steps with tool calls are tool-request steps; assistant steps followed by a graceful turn boundary are final answers; unfinished or crashed spans can be labeled running, aborted, or unknown.

Rules of Thumb

  • Do not ask "which user message owns this assistant message?" Ask "which turn contains this model step?"
  • Do not switch to arbitrary assistant entries. Switch to stable context boundaries.
  • Do allow fork from arbitrary entries, but make the target semantics explicit.
  • Do not export isolated assistant text by default when it belongs to a tool-request/progress step. Export a final answer or an entire turn.
  • Do not make provider replay drive UI semantics. Provider replay is for continuity; derived turn/model-step views are for app logic.

Rust Invariant Audit

Rust should enforce the facts that protect the append-only transcript tree. It should not enforce UI labels such as progress versus final answer.

  • history.rewind and history.switch-style source movement remain boundary-only: empty root, TurnFinished, or CompactionSummary.
  • history.fork is idle-only and accepts only the current active TurnFinished or CompactionSummary target while Git workspaces are copied from the live checkout.
  • Compaction remains boundary-only and appends a compaction summary root inside the same session tree/forest.
  • Provider replay stays sidecar-only and is stored on transcript rows without becoming visible transcript content.
  • Rust tests cover the shared fork/rewind boundary rule: non-boundaries are rejected and fork requires an idle source.

Implementation Order

  1. Add a turn-view helper and fixture tests in packages/web.
  2. Update export to consume turn/model-step views instead of nearest-user pairing.
  3. Update fork/switch picker labels and filters to use derived phases.
  4. Polish transcript rendering for tool-request/progress steps versus final answer steps.
  5. Audit existing Rust tests/code for the structural invariants above; add narrow Rust tests only for gaps.
  6. Optionally add provider-local stop metadata tests only where they affect error handling or retry semantics.
  7. Add one websocket smoke that creates a multi-step tool turn, exports it, forks from a tool-request step, and switches to the final boundary.
