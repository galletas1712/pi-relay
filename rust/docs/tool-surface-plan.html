Provider Tool Surface Plan

Goal: stop treating every tool as a provider-neutral JSON function, while keeping pi-relay small, local-first, and cache-friendly.

Summary

Status update (2026-05-14): the live posture for shell diverges from the original plan below. The bash tool is now a single custom function tool named bash exposed identically to both providers — pi-relay does not use Anthropic's native bash_20250124 wrapper. The runtime is still a one-shot bash -lc per call (no PersistentTerminal yet); the model-visible contract was aligned with that reality rather than the other way around. The edit tools remain divergent (apply_patch on OpenAI, text_editor_20250728 on Anthropic) because their schemas are semantically rich enough to justify the provider-native form. See design-decisions.md § "Provider Tool Surfaces Diverge Only When Semantics Justify It" for the rationale. The rest of this document still describes the long-term direction (persistent terminal, worktree scoping, etc.).

The main abstraction change should be simple: split what the model sees from who executes the tool. Today agent_vocab::ToolDefinition is doing too much: it is simultaneously a model schema, a local dispatch key, and a provider request shape. That is why local bash, provider-hosted web_search, Anthropic server tools, future MCP, and OpenAI freeform/custom tools all feel like exceptions.

The updated direction is even narrower: do not force OpenAI and Anthropic to share a fake common tool set such as bash/read/write/edit. Each provider should expose the coding tools it is trained to use. Underneath those provider-native tools, pi-relay can still share the same local runtimes: a persistent terminal, a patch applier, and a text-editor filesystem adapter.

The new design should keep provider-specific ToolProfile rendering in agent-provider, and keep execution runtimes in agent-tools. Runtime dispatch only handles tools whose execution owner is pi-relay. Provider-hosted tools are stored and rendered as transcript events, but never become pending local actions.

What The APIs Actually Expose

OpenAI Responses

OpenAI's current docs say Responses tools include built-in tools, function calling, tool search, and remote MCP servers. New web search integrations should use { "type": "web_search" }; responses include web_search_call output items and citations. There is no separate top-level OpenAI web_fetch tool like Anthropic's. Instead, the OpenAI web search tool can emit actions such as search, open_page, and find_in_page; Codex models those as WebSearchAction::Search, WebSearchAction::OpenPage, and WebSearchAction::FindInPage. Function tools are normal JSON schema tools. Custom tools can use context-free grammars such as Lark or regex, which is the path Codex uses for token-efficient freeform payloads such as patches.

For the OpenAI coding profile, follow the OpenAI/Codex harness shape: shell/search commands are used for reading and inspecting the repository, while apply_patch is the edit primitive. Codex does not expose a general workspace read/write function in the core coding surface; file inspection happens through shell commands such as rg, sed, ls, and find. MCP resource reads are a separate MCP surface, not the normal filesystem read path.

We should not enable hosted shell because it is a cloud sandbox. The OpenAI coding profile should expose only shell as the model-visible terminal tool, backed by pi-relay's persistent terminal runtime and scoped to the session worktree. Treat local_shell as a historical compatibility detail from Codex internals, not as the target pi-relay surface.

Anthropic Messages

Anthropic's docs make the execution boundary explicit. Client tools produce tool_use blocks, our application executes them, and we send back tool_result. Server tools such as web_search, web_fetch, code_execution, and tool_search run on Anthropic infrastructure; the API response includes server_tool_use and tool result blocks, but pi-relay does not execute those tools locally.

The Anthropic coding profile should use Anthropic-schema client tools directly: bash_20250124 for shell and text_editor_20250728 for file viewing/creation/editing. These are not server tools: Anthropic defines the schema, but pi-relay still executes the operation. The adapter maps Anthropic's native command shapes into the same persistent terminal and filesystem runtimes used by the OpenAI profile.

The takeaway is that many agent systems still define custom JSON tools because the abstraction is portable and easy to validate. That is acceptable for small arguments such as pattern, path, or session_id. It is expensive for large text payloads such as full file contents, shell heredocs, or patches.

pi-relay Policy

  • Use provider-native coding tools wherever the provider has a first-class client tool.
  • Keep apply_patch always available for OpenAI, with a native or freeform/custom wire shape.
  • Keep grep as a custom JSON tool because its inputs are small and its output can be bounded aggressively.
  • Do not expose generic write_file(content) or patch-as-JSON-string tools in the normal coding profile.
  • Optimize tool result sizes as much as tool call syntax: bounded search/read output saves more context than shaving a few JSON tokens from small calls.
  • Keep the provider replay model structured. Token-efficient tool calls are not worth losing clean switch, compaction, and retry semantics.

Tool Taxonomy

Local Client Tools

pi-relay executes these in the selected session worktree and returns a result to the model. The model-visible name does not need to be shared across providers. Examples: OpenAI shell, OpenAI apply_patch, Anthropic bash_20250124, Anthropic text_editor_20250728, future local MCP client tools.

Keep This is the core local substrate, but provider profiles choose the visible schema.

Provider-Hosted Tools

The provider executes these inside the model request. pi-relay enables them, parses resulting events, stores provider replay, and renders them in the UI. Examples: OpenAI web_search, Anthropic web_search_20250305, Anthropic web_fetch_20250910.

Add No local pending action is created.

Provider-Native Client Tools

The provider owns the schema, but pi-relay executes the operation. Examples: OpenAI shell, OpenAI apply_patch, Anthropic bash_20250124, and Anthropic text_editor_20250728.

Use now This should be the main coding surface instead of a provider-neutral bash/read/write/edit facade.

Custom Tools

Tools we inject that are not provider built-ins. They may be JSON-schema tools on both providers or OpenAI-only freeform/custom grammar tools. This is where an OpenAI apply_patch grammar or a pi-relay-specific file operation substrate belongs.

Design now Keep this extensible without making every tool provider-specific.

MCP Tools

Two modes exist: pi-relay can run a local MCP client and expose tools as local client tools, or the provider can connect to remote MCP servers directly. For personal use, local MCP client mode is the better default because it supports stdio/local servers and keeps credentials under pi-relay control.

Phase later Design for it, do not implement first.

Cloud Sandbox Tools

Provider code execution, hosted shell containers, and computer-use sandboxes are powerful but not aligned with the current goal. The user explicitly does not want cloud sandboxes surfaced here.

Exclude Do not enable by default.

Provider-Native Client Tools

Provider-native client tools should become the normal coding surface. The provider owns the model-visible schema and training surface; pi-relay owns execution. This avoids making Claude pretend it has OpenAI tools, and avoids making OpenAI pretend it has Anthropic's text editor.

The shared part is below the model boundary: terminal execution, patch application, filesystem editing, truncation, timeout handling, worktree scoping, and transcript result storage. The session FSM should still only see "local tool requested" and "tool result appended"; it should not need to know whether the visible request was OpenAI shell, Anthropic bash, OpenAI apply_patch, or Anthropic text editor.

How To Wire Them

The adapter should be provider-specific only at the request/response boundary. Internally, all provider-native client calls should become a local action over one of a few runtime capabilities.

Provider response item
  -> provider parser recognizes native client call
  -> ProviderToolCall { profile, native_kind, call_id, raw_input }
  -> provider adapter resolves native_kind to LocalRuntimeKind
  -> adapter converts raw_input into a runtime command
  -> existing session action lifecycle runs
  -> adapter converts runtime output into provider-native tool result

This needs three small abstraction additions:

  • ProviderToolProfile: OpenAiCoding or AnthropicCoding, selected by the session's provider/model.
  • ProviderNativeClientKind: OpenAiShell, OpenAiApplyPatch, AnthropicBash, AnthropicTextEditor, etc.
  • LocalRuntimeKind: PersistentTerminal, WorkspaceFileRuntime, or WorkspaceSearchRuntime.

Do not make agent-session know OpenAI or Anthropic schemas. The FSM can store an opaque local action with a provider call id and normalized result lifecycle; provider replay keeps the exact wire item needed to resume the model.

Shared Runtime Layers

The provider-native adapters should stay thin. OpenAiShell and AnthropicBash should both call into one terminal runtime. OpenAiApplyPatch and AnthropicTextEditor should both call into one workspace-file substrate, even though their model-visible commands are different. grep is a separate read-only search runtime backed by ripgrep.

The important distinction is adapter versus runtime. An adapter translates a provider-native or custom model-visible shape into a local command. A runtime owns behavior that must be identical across providers: path safety, cwd/worktree selection, file mutation semantics, terminal persistence, search limits, truncation, and error formatting.

Provider Output Parsing

Provider outputs should be normalized into two categories:

  • Local tool requests: create a pending action in agent-session, then agent-daemon dispatches through agent-tools.
  • Provider-hosted tool events: store in provider_replay, surface to the UI as transcript metadata, and keep them in replay context for future provider calls.
ProviderOutputItem
  AssistantText
  LocalToolCall { id, provider_native_kind, raw_input }
  HostedToolUse { id, provider, name, input, result, status }
  Reasoning
  UnknownReplayable { raw }

This avoids pretending that Anthropic server_tool_use or OpenAI web_search_call is a local action. It also keeps switch, compaction, and retry semantics clean because provider-hosted events remain append-only transcript/replay data.

Flow Diagrams

Local Tool Flow

User message
  -> provider request with provider-native coding profile
  -> model emits OpenAI shell/apply_patch or Anthropic bash/text_editor
  -> provider parser records a local pending action with the raw provider call id
  -> agent-daemon dispatches the matching agent-tools runtime
  -> tool result is appended
  -> provider adapter renders the provider-native tool result on continuation

Provider-Hosted Tool Flow

User message
  -> provider request with hosted tool spec
  -> provider/model runs hosted web_search or web_fetch internally
  -> provider response includes hosted tool events and final text
  -> pi-relay stores raw replay items
  -> UI renders search/fetch events
  -> no local pending action is created

MCP Later

Local MCP mode:
  RuntimeRegistry discovers MCP tool
  -> render through provider-specific MCP/tool-search profile
  -> model emits local tool call
  -> pi-relay calls MCP server

Provider-hosted MCP mode:
  Provider profile renders provider MCP toolset
  -> provider calls remote MCP server
  -> pi-relay observes hosted MCP events only

Implementation Plan

Phase 1: Add Provider Coding Profiles

  • Add ProviderToolProfile::OpenAiCoding and ProviderToolProfile::AnthropicCoding in agent-provider.
  • Stop treating bash/read/write/edit as the provider-neutral model contract for new sessions.
  • Render OpenAI as shell plus apply_patch plus custom grep plus provider-hosted web_search.
  • Render Anthropic as bash_20250124, text_editor_20250728, custom grep, provider-hosted web_search_20250305, and provider-hosted web_fetch_20250910.
  • Keep deterministic ordering and cache markers stable inside the renderers.

Phase 2: Build Shared Local Runtimes

  • Add PersistentTerminal per session/worktree. OpenAI shell and Anthropic bash both use it.
  • Add one WorkspaceFileRuntime. OpenAI apply_patch and Anthropic text_editor_20250728 both use it.
  • Add one WorkspaceSearchRuntime. The custom grep tool invokes ripgrep directly instead of going through shell.
  • Put patch application, exact replacement, insert, create, delete, and file viewing behind the same path-scoped workspace file layer.
  • Keep path scoping, output truncation, timeout, command lifecycle behavior, and file mutation semantics in these runtimes rather than in provider code.

Phase 3: Parse Native Calls And Hosted Events

  • OpenAI: parse shell calls into PersistentTerminal actions, apply_patch calls into WorkspaceFileRuntime actions, and grep calls into WorkspaceSearchRuntime actions.
  • OpenAI: parse and persist web_search_call output items, including action details and citations where present.
  • Anthropic: parse bash_20250124 calls into PersistentTerminal actions, text_editor_20250728 calls into WorkspaceFileRuntime actions, and grep calls into WorkspaceSearchRuntime actions.
  • Anthropic: parse and persist server_tool_use, web_search_tool_result, and web_fetch_tool_result blocks.
  • Do not turn hosted events into ToolCallStarted or local pending actions.
  • Expose hosted events over WebSocket RPC as transcript decorations so the UI can render them like tool cards.

Phase 4: Retire Legacy Generic Tools

  • Remove provider exposure for generic JSON bash/read/write/edit once native profiles pass live tests.
  • Keep the underlying code only if a runtime still needs it; otherwise delete the unused handlers.
  • Keep ToolDefinition only where legacy transcript replay requires it. New provider calls should go through native profiles.
  • Document that file reading on OpenAI is shell-based, while file reading on Anthropic is text-editor-based.

Phase 5: MCP And Deferral

  • Add local MCP client tool discovery when there is a real MCP server to support.
  • Expose local MCP tools through provider-specific renderers only after the first native coding profiles are stable.
  • Add OpenAI tool_search and Anthropic defer_loading only after the tool catalog is large enough to justify it.
  • Do not add provider-hosted MCP first. It only supports public remote servers and would complicate credential boundaries.

pi-relay Custom Grep Schema

grep is intentionally custom because neither upstream coding surface gives us a dedicated ripgrep tool. It should be read-only, strict, bounded, and backed by rg directly rather than by shell string interpolation.

{
  "name": "grep",
  "description": "Search files in the current workspace with ripgrep.",
  "parameters": {
    "type": "object",
    "additionalProperties": false,
    "properties": {
      "pattern": {
        "type": "string",
        "description": "Regular expression to search for."
      },
      "path": {
        "type": "string",
        "description": "Optional file or directory path relative to the workspace root."
      },
      "include": {
        "type": "array",
        "items": { "type": "string" },
        "description": "Optional glob allow-list, for example ['*.rs', '*.ts']."
      },
      "exclude": {
        "type": "array",
        "items": { "type": "string" },
        "description": "Optional glob deny-list."
      },
      "context": {
        "type": "integer",
        "description": "Number of context lines before and after each match."
      },
      "max_matches": {
        "type": "integer",
        "description": "Maximum number of matching lines to return."
      },
      "case_sensitive": {
        "type": "boolean",
        "description": "Whether matching should be case-sensitive."
      }
    },
    "required": ["pattern"]
  }
}

This is the logical schema. The OpenAI strict renderer may need to transform optional fields into nullable required fields to satisfy Structured Outputs constraints; the runtime should treat null the same as omission. Runtime defaults should stay stable: path defaults to the workspace root, context defaults to 0, max_matches defaults to a modest cap such as 100, and omitted case_sensitive uses ripgrep's normal smart-case behavior.

Tool Spec Knobs

Tool Effort Defaults

Provider-native tools should follow the same bias as model reasoning: default to high-quality retrieval and generous inspection. Avoid provider-visible max_* fields by default. Most of them are budgets or truncation controls, not quality knobs, and they can create surprising failures or hidden information loss.

The model does not configure these values. pi-relay sets them when rendering the provider request. The model can choose whether to use an enabled tool and can choose the tool input, such as a search query or URL. Optional provider max_* fields should be added only for explicit workflow budgets, not as part of the normal high-effort profile.

ProviderToolHigh-effort defaultWhy
OpenAI web_search search_context_size: "high" Matches the intent of xhigh reasoning: let the hosted search tool provide richer context when the model decides web evidence matters.
Anthropic web_search_20250305 omit max_uses by default Let Claude perform the searches it needs. Add max_uses only for a workflow that explicitly wants a budget.
Anthropic web_fetch_20250910 citations.enabled: true; omit max_uses and max_content_tokens by default Fetch should be useful for substantial docs/pages. Add provider limits only for explicit budgets or truncation policies.
Anthropic text_editor_20250728 omit max_characters by default Do not silently truncate file views at the provider-tool level. Use targeted ranges and local runtime safety limits instead.

Research Mode

OpenAI deep research should be a separate provider profile, not a hidden mutation of the normal coding profile. OpenAI exposes deep research through dedicated models such as o3-deep-research and o4-mini-deep-research in the Responses API. A deep research request must include at least one data source, such as web_search, remote MCP, or file search. It can also include code_interpreter, but that is a cloud sandbox and should stay out of the default coding profile.

For deep research, pi-relay should set request-level research policy explicitly:

OpenAIResearchProfile {
  model: "o3-deep-research" | "o4-mini-deep-research",
  tools: [
    WebSearch { search_context_size: "high", external_web_access: true },
    // optional later: remote MCP or file_search
    // optional only if explicitly enabled: code_interpreter
  ],
  background: true,
  max_tool_calls: optional_explicit_research_budget,
}

max_tool_calls is different from a tool's own quality setting. It is a total request budget for deep research tool calls such as web search, MCP, and code interpreter. The model still decides how to spend that budget. Similarly, Anthropic max_uses, max_characters, and max_content_tokens are guardrail/truncation controls. Omit them by default and add them only when the user explicitly wants a bounded research run or a workflow has a known cap.

OpenAI Responses

OpenAI tool knobs depend on the tool kind. Built-in tools have small fixed specs. Function and custom tools expose richer model-visible metadata. The first implementation uses a Codex-compatible function tool named shell, OpenAI hosted web_search, a custom-function grep, and Codex-style custom/freeform apply_patch. The ChatGPT/Codex subscription endpoint rejected the platform { "type": "shell" } shape during validation, so pi-relay keeps the model-visible name shell but renders it as a function tool for this transport.

Invariants To Enforce

  • A provider-hosted tool must never create an ActionRequested row or local pending action.
  • A provider-native client call must resolve to a known local runtime before it can enter the session FSM.
  • The OpenAI coding profile must always include apply_patch; only its wire shape may vary.
  • Provider replay must preserve hosted tool events exactly enough to reconstruct the next stateless provider request.
  • Provider tool lists must be deterministically sorted, and cache-affecting tool descriptions must not drift per turn.
  • Provider-native tool effort defaults must be high and stable without unnecessary provider-visible max_* caps. OpenAI web search uses high context; Anthropic web search/fetch and text editor omit optional caps unless a workflow explicitly asks for a budget or truncation policy.
  • Provider-specific tool versions belong in provider rendering config, not in session state.
  • A session's selected provider/model determines its coding tool profile. Do not change the provider-visible tool profile mid-session.
  • Cloud sandbox tools remain disabled unless explicitly added later.

Open Questions

  • For OpenAI apply_patch, the first implementation uses the exact Codex-style freeform custom grammar shape. A native provider item shape can be evaluated later without changing the local patch runtime.
  • For OpenAI shell, the first implementation renders a function tool named shell because the ChatGPT/Codex subscription endpoint rejected the platform { "type": "shell" } shape with Unsupported tool type: shell. The target model-visible tool remains shell; do not design around deprecated local_shell. The provider still parses shell_call defensively in case a future transport supports it.
  • Should exact URL fetch exist as a pi-relay local tool? Anthropic has native server web_fetch. OpenAI covers page opening and in-page search as hosted web_search actions rather than a separate web_fetch tool. A future local web_fetch runtime may still be useful for provider parity or authenticated/local-network fetches, but it should not block the first profile.
