Implementation plan

Provider replay continuity plan

This plan moves pi-relay from a chat-shaped provider model to a small provider-neutral replay model. The goal is to support OpenAI Responses and Anthropic Messages faithfully while keeping agent-provider stateless, agent-session provider-agnostic, and the web UI focused on human-visible transcript items.

Target architecture

The canonical session remains an append-only history tree in Postgres. Each node can carry human-visible transcript data and optional provider replay data. Provider adapters serialize that replay data back to OpenAI or Anthropic without exposing provider-specific ceremony to the rest of the system.

Session tree append-only source of truth
Model context branch + compaction roots
Replay sidecar opaque provider items
Provider adapter OpenAI or Anthropic wire format
Model output new records appended

Non-goal

Do not build a broad provider abstraction framework. We only need enough surface for OpenAI and Anthropic, with room to add custom/freeform OpenAI tools later.

Cloud sandboxes, remote sessions, and managed cloud agents are out of scope. Remote provider tools such as web search are in scope because they are still tool-loop replay records.

Core data shape

Add one small sidecar type for provider replay. The sidecar is provider-neutral at the top level, but intentionally stores raw provider payloads for exact replay. 

struct ProviderReplayItem {
    provider: ProviderKind,
    raw_json: String,
}

No provider replay variant belongs in AssistantItem. Visible transcript data and opaque provider continuation data stay aligned by transcript entry, but are not the same abstraction.

Provider request shape

ModelRequest passes prepared transcript entries, not raw store nodes. Each entry has the semantic item plus optional provider replay sidecar. 

struct ModelRequest {
    model: String,
    prompt: PromptSections,
    transcript: Vec<ModelTranscriptEntry>,
    tools: Vec<ToolDefinition>,
    max_tokens: Option<u32>,
    prompt_cache_key: Option<String>,
}

struct ModelTranscriptEntry {
    item: TranscriptItem,
    provider_replay: Vec<ProviderReplayItem>,
}

The provider serializes replay items when present and falls back to semantic transcript items when replay is absent, which keeps old and compacted history replayable.

Phase plan

  1. Phase 1: Add replay records without changing behavior Add ProviderReplayItem, extend provider parser tests, and store replay sidecars for new outputs. Existing UI rendering remains intact because replay data is not an assistant item.
  2. Phase 2: Make OpenAI Responses stateless-correct Set include: ["reasoning.encrypted_content"] for Responses requests, parse reasoning, message.phase, function_call, and tool output items, and serialize replay records back into input[]. Do not use previous_response_id for correctness.
  3. Phase 3: Make Anthropic thinking/tool replay faithful Parse and preserve thinking, redacted_thinking, signature, tool_use, and tool_result shapes. Add Anthropic request serialization from replay records. Keep tool_choice:auto when extended thinking is enabled.
  4. Phase 4: Update session context building Teach agent-session to build model context from the selected branch plus compaction roots. Fork/switch should carry replay records with the branch. Compaction should summarize completed history but leave active tool-loop replay records intact.
  5. Phase 5: Add custom/freeform tool support for OpenAI Responses Represent tool definitions with input modes: JSON schema or freeform grammar. Initially use JSON for existing tools and add freeform only for patch/file-edit-like tools where it clearly reduces token overhead.
  6. Phase 6: Add remote/provider tool replay Add first-class replay handling for provider-run tools such as web search. These should render as tools in the UI, but their raw provider call/result payloads should stay in the replay lane.
  7. Phase 7: Tighten UI and inspector Show visible messages, tool calls, tool results, turn outcomes, compaction roots, and branches. Hide opaque reasoning/thinking by default. In the inspector, show counts and provider kinds for replay records so debugging is possible without leaking useless blobs into the transcript.
  8. Phase 8: Remove compatibility glue Remove the OpenAI Chat Completions path and any tests that assert the older Chat-Completions-shaped behavior. OpenAI should be Responses-only.

Implementation notes

  • OpenAI is now implemented as a Responses-only provider through ChatGPT/Codex subscription auth. The provider keeps prompt_cache_key but omits prompt_cache_retention because pi-relay does not use the plain OpenAI API-key transport. OpenAI/Codex requests also omit max_output_tokens; pi-relay should not impose a token cap, and the Codex Responses endpoint rejects that field.
  • Provider replay now lives in StoredTranscriptEntry.provider_replay and transcript_entries.provider_replay. A migration lifts legacy provider_replay_record assistant items into that sidecar, keeping the append-only transcript tree intact while removing replay blobs from visible assistant data.
  • Anthropic requests enable extended thinking and preserve complete raw thinking, redacted_thinking, text, and tool_use blocks for round-trip continuity. The UI hides these opaque records by default. The stable system prefix is sent as a cached system block with a 1-hour cache_control marker; dynamic context is an uncached suffix.
  • Custom/freeform OpenAI tools and first-class hosted-tool UI are still separate additions. The existing registry only exposes JSON-schema local tools, so this pass avoids adding an unused tool-definition abstraction until a concrete freeform tool is introduced.

OpenAI details

  • Responses request policy Always use store:false, parallel_tool_calls:true, and include:["reasoning.encrypted_content"] when the model/output mode can produce reasoning continuity.
  • Replay serializer Serialize prior user messages, provider replay sidecars, assistant messages, function/custom tool calls, and matching tool outputs into Responses input[].
  • Freeform tools Defer ToolInputMode::FreeformGrammar until there is a concrete patch/file-edit tool that needs raw grammar input. The current local tools remain JSON-schema tools.
  • Chat Completions Removed from the OpenAI provider path. Reintroduce only as a deliberately lossy provider adapter if there is a concrete need.

Anthropic details

  • Thinking preservation Preserve complete thinking blocks with signature and complete redacted_thinking blocks during active tool trajectories.
  • Tool pairing Reject invalid tool chains before provider calls: no orphaned tool_result, no duplicate tool_use IDs, and no missing result for a completed local tool.
  • Prompt caching Cache the stable prefix with cache_control. Because Anthropic caches the prompt prefix in tools, system, messages order, a cache marker on the stable system block also covers the stable local tool definitions that precede it.
  • Streaming fallback If a stream fails after tool calls have appeared, discard partial replay sidecars for that attempt unless matching tool results were fully persisted.

Storage plan

Keep Postgres as the only source of truth. Avoid a new generic storage trait. The schema should make visible transcript state and provider replay state easy to query independently.

Option Shape Pros Cons Recommendation
Inline transcript JSONB Add replay records inside the visible transcript item JSON. Small migration, entry-local, easy append. Blurs semantic transcript data with opaque provider continuation data. Rejected.
Sibling JSONB column transcript_entries.provider_replay jsonb not null default '[]' Clear separation without a join. UI can ignore by default. Requires migration and store changes. Implemented.
Separate table provider_replay_records(entry_id, ordinal, provider, raw) Best query/debug story. More schema and write complexity. Defer until needed.

Compaction, fork, switch

  • Compaction Compaction creates a new context root inside the same session. It summarizes completed visible history and completed replay history, but must not summarize the open tool loop.
  • Fork Fork can branch from any transcript point. The new branch inherits replay records along the path to that point. If the point is inside an open provider trajectory, only valid ancestor records are included.
  • Switch Switch is a source mutation and remains idle-only. It selects another leaf/context root inside the same session and must update both visible context and replay context.
  • Rewind Rewind can stay removed as a separate concept. UI "edit old message" is branch creation from the selected point plus composer prefill.

Lifecycle rules

  • During active turn Follow-ups can queue, steers can be promoted before consumption, fork is allowed from a stable prior point, switch/compact are blocked until idle or interrupt.
  • During tool execution The tool call and eventual output must be atomically persisted enough that daemon death cannot leave a provider-invalid transcript.
  • After daemon restart Recovery scans unfinished actions, appends interrupted/crashed tool results where needed, and validates provider replay chains before allowing new model calls.
  • On provider error Do not patch historical provider records. Append a turn outcome/error item and leave the history tree inspectable.

Risks

  • Over-modeling A large generic provider framework would make this worse. Keep the enum small and let providers own raw payload translation.
  • Replay/UI leakage Opaque reasoning must not come back as visible transcript rows.
  • Invalid active trajectories Partial streaming outputs and daemon death can leave unmatched tool calls unless writes are ordered carefully.
  • Provider drift OpenAI and Anthropic fields can change. Preserve raw records so new fields survive before we add first-class types.

Definition of done

  • OpenAI Responses Done: real Codex Responses turns cover text, replay-preserving continuation, local tool use, and daemon restart recovery after a killed in-flight tool action.
  • Anthropic Done: real Anthropic Messages turns preserve thinking signatures across text, continuation, and local tool-use loops.
  • History operations Done at the WebSocket/storage layer: fork while active, switch blocking while active, fork continuation, manual compaction, and switching between a compaction root and pre-compaction branch all preserve visible transcript and replay records.
  • UI No opaque reasoning/thinking rows appear in normal chat, while tools and turn outcomes still render clearly.
  • Simplicity No generic storage trait, no extra provider abstraction layer, and no broad compatibility fallback paths beyond the two supported providers.
