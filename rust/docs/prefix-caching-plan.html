Prefix Caching Plan

Goal: make provider requests cache-friendly by construction, with request-shape tests and real usage metrics proving cache hits.

Summary

pi-relay uses OpenAI through Codex subscription auth only. The OpenAI provider should therefore match the Codex harness shape: ChatGPT bearer token, Codex backend, prompt_cache_key, store: false, priority service tier, and no plain API-key request path.

The cache boundary is now provider-owned and testable. OpenAI keeps dynamic context out of the stable instructions prefix and uses a deterministic cache cohort key. Anthropic marks stable tools, stable system prompt, and the latest eligible transcript block with cache breakpoints, following the same broad shape as Claude Code.

Provider Rules

ProviderWhat the API wantspi-relay target shape
OpenAI Responses Prompt caching is automatic for eligible prompts. Exact prefix matches matter. prompt_cache_key helps route related requests. Codex subscription transport should not send unsupported plain-API-only fields. Stable instructions and stable sorted tools first; dynamic cwd/worktree/session state later in input. Use a deterministic cache cohort key, not a magic global string. Log usage.input_tokens_details.cached_tokens.
Anthropic Messages Caches are controlled with cache_control breakpoints. Cache order is tools → system → messages. There are up to 4 breakpoints, and the automatic backward walk looks back ~20 content blocks per breakpoint. 5m writes cost 1.25x base input; 1h writes cost 2x; reads cost 0.1x. Mark the stable system block (1h TTL) and the latest eligible message block (5m TTL). For long transcripts also mark a deeper history block so coverage extends past the 20-block lookback. No per-tool marker — the stable-system breakpoint covers tools transitively. Preserve thinking blocks in provider replay. Log cache read/write usage.

Current Diagnosis

OpenAI

  • done agent-provider/src/openai.rs uses the Codex subscription backend only; the old plain API-key auth split is gone.
  • done responses_body renders instructions as stable prefix only. Dynamic context is the first synthetic input item, followed by transcript items.
  • done Default cache key is the pi-relay session id (matching Codex CLI’s prompt_cache_key = thread_id.to_string()): one routing bucket per session, well under OpenAI’s ~15 RPM-per-shard ceiling while still maximizing in-session prefix reuse. An explicit ProviderConfig.prompt_cache.key override still wins.
  • done The HTTP envelope mirrors Codex CLI byte-for-byte: originator: codex_cli_rs, User-Agent: codex_cli_rs/{ver} ({os}; {arch}), x-openai-internal-codex-residency: us, x-codex-installation-id (from ~/.codex/installation_id), x-codex-window-id (per-process UUID), and the full session_id/session-id/thread_id/thread-id/x-client-request-id routing pair. This matches build_responses_identity_headers+build_session_headers+default_headers in the Codex CLI source.
  • done Provider usage/cache metrics are parsed into ModelResponse.usage and persisted on model action results.

Anthropic

  • done anthropic_system_blocks marks the stable system block with cache_control: { type: "ephemeral", ttl: "1h" }.
  • good Provider replay stores thinking and redacted_thinking blocks unchanged and replays them before normalized visible text/tool calls.
  • done A 5-minute (default ephemeral) cache breakpoint is added to the latest eligible transcript block. 5m is correct here because the marker is regenerated each turn — 1h writes cost 2x base input vs 1.25x for 5m, so the long-TTL premium would be wasted.
  • done When the transcript exceeds Anthropic's ~20-block lookback window, an additional 5m breakpoint is placed deeper in the history (~18 cacheable-blocks back from the tail) so older prefixes remain discoverable past the automatic lookback ceiling.
  • done Tool schemas are rendered in deterministic sorted order. No per-tool cache_control: the stable-system breakpoint already covers tools via Anthropic's tools → system → messages cumulative prefix hash, so a tools-level marker would burn a breakpoint slot for zero gain.
  • done The Claude-Code attribution fingerprint is derived from the stable system prompt (not the first user message), so two sessions with the same global prompt produce the same system[0] bytes and can share the cached stable-system prefix across sessions.
  • done thinking: { type: "adaptive" } is hard-coded with a comment: Anthropic invalidates message-content cache on any thinking change, so this must not become a per-request toggle.
  • done cache_read_input_tokens and cache_creation_input_tokens are parsed into provider usage.

Target Request Shapes

// OpenAI Responses
{
  model,
  instructions: stable_system_prompt_only,
  tools: stable_sorted_tools,
  input: [
    { role: "user", content: [{ type: "input_text", text: dynamic_context_label }] },
    ...transcript_items
  ],
  prompt_cache_key: cache_cohort_key,
  service_tier: "priority",
  parallel_tool_calls: true,
  store: false
}
// Anthropic Messages
{
  tools: [
    ...stable_sorted_tools,
    // no per-tool cache_control: covered by the stable-system breakpoint
    // via Anthropic's tools -> system -> messages cumulative prefix hash
  ],
  system: [
    { type: "text", text: attribution_header },                                           // fingerprinted off stable_prefix, cross-session-stable
    { type: "text", text: stable_system_prompt, cache_control: { type: "ephemeral", ttl: "1h" } },
    { type: "text", text: dynamic_context_label }                                          // uncached suffix
  ],
  messages: [
    ...transcript_messages_with_provider_replay,
    // latest eligible message block gets cache_control: { type: "ephemeral" }  (5m default)
    // PLUS: when total cacheable blocks > ~18, an additional 5m marker
    //       roughly 18 blocks back from the tail, to extend coverage past
    //       Anthropic's 20-block automatic lookback ceiling
  ],
  thinking: { type: "adaptive" },
  output_config: { effort }
}

Implementation Plan

Phase 1: Keep cache policy at the provider boundary

  1. done Do not add a new cache-planning abstraction yet. The rules are small enough to live in the provider serializers.
  2. Keep it out of agent-session. Cache policy is provider serialization/runtime behavior, not transcript semantics.
  3. done Derive the OpenAI default cache cohort from model, stable prompt, and stable tool schema. Do not include current user input, cwd, active leaf id, or timestamps.
  4. done Allow explicit override from ProviderConfig.prompt_cache for experiments, but make the default good.

Phase 2: Fix OpenAI request shape

  1. done Render instructions as the stable system prompt only.
  2. done Move dynamic context into a synthetic first input item, after the stable request prefix. It stays plain and timestamp-free.
  3. done Omit prompt_cache_retention; pi-relay does not use OpenAI API-key auth.
  4. done Parse response.completed.response.usage and store/log cached_tokens.
  5. done Add request-body tests: changing cwd/transcript changes only dynamic input/transcript, not instructions, tools, or cache key.

Phase 3: Fix Anthropic request shape

  1. done Keep the stable system cache_control marker.
  2. done Add exactly one message-level cache_control marker to the latest eligible content block, similar to Claude Code’s messagesToMessageParams flow.
  3. done Do not add cache_control to thinking blocks or non-cacheable content blocks.
  4. done Keep tool definitions stable with deterministic ordering and a cache marker on the last stable tool.
  5. done Parse response usage and store/log cache_creation_input_tokens, cache_read_input_tokens, and input/output tokens.
  6. done Add tests for stable system, sorted tools, latest-message cache marker, thinking/replay preservation, and usage fields.

Phase 4: Add observability

  1. done Extend ModelResponse with ProviderUsage. Keep the fields provider-neutral but lossless enough for cache metrics.
  2. done Persist usage on the model action row, not as transcript content.
  3. done Stop there for now. The model action result is enough to inspect cache behavior when debugging without adding new UI, logs, or cache-specific commands.

Phase 5: Real verification

  1. OpenAI: send two similar Responses requests through the daemon in one session. Assert the second response reports non-zero cached_tokens once the prompt is above the minimum.
  2. Anthropic: send two similar Messages requests through the daemon. Assert first request creates cache tokens and the second reads cache tokens.
  3. Run a multi-tool turn and verify Anthropic thinking/replay blocks are preserved while cache metrics remain sensible.
  4. Run switch/compact after cached turns and verify no cache metadata appears in the visible transcript tree.

Design Choices

  • Cache policy belongs in agent-provider/agent-daemon, not in agent-session. Sessions store semantic transcript facts; providers decide how to serialize those facts efficiently.
  • Usage metrics belong with action/runtime records, not transcript entries. They describe a provider call, not a user-visible conversation event.
  • Dynamic context should be late and minimal. Worktree/cwd matter, timestamps and UI state do not.
  • Tool schema stability is as important as prompt stability. Adding remote web search is fine, but it should be a deliberate tool-set change because tools are part of the cacheable prefix.
  • No token cap is required for caching. We only need to structure the prompt so repeated prefixes are actually reusable.
